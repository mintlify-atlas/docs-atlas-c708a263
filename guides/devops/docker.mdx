## What is Docker?

Docker is a platform for developing, shipping, and running applications in containers. Containers package your application with all its dependencies, ensuring it runs consistently across different environments.

![Docker Architecture](https://assets.bytebytego.com/diagrams/0414-how-does-docker-work.png)

## Docker Architecture

There are 3 components in Docker architecture:

### Docker Client

The docker client talks to the Docker daemon. It's the primary way users interact with Docker through commands like `docker build`, `docker pull`, and `docker run`.

### Docker Host

The Docker daemon listens for Docker API requests and manages Docker objects such as:
- Images
- Containers
- Networks
- Volumes

### Docker Registry

A Docker registry stores Docker images. Docker Hub is a public registry that anyone can use, but you can also run private registries for your organization.

## How Docker Commands Work

Let's examine what happens when you run common Docker commands:

<Steps>
  <Step title="docker pull">
    Docker daemon contacts the registry and downloads the specified image to the local host
  </Step>
  
  <Step title="docker build">
    Docker reads the Dockerfile, executes instructions, and creates a new image
  </Step>
  
  <Step title="docker run">
    Docker pulls the image (if needed), creates a container, allocates filesystem, sets up networking, and starts the container
  </Step>
</Steps>

### Docker Run Deep Dive

When you execute `docker run`, Docker performs these operations:

1. **Pulls the image** from the registry (if not available locally)
2. **Creates a new container** from the image
3. **Allocates a read-write filesystem** to the container
4. **Creates a network interface** to connect the container to the default network
5. **Starts the container** and executes the specified command

## Essential Docker Concepts

![Docker Concepts](https://assets.bytebytego.com/diagrams/0012-8-must-know-docker-concepts.png)

### 1. Dockerfile

A Dockerfile contains instructions to build a Docker image by specifying the base image, dependencies, and run commands.

<CodeGroup>
```dockerfile Basic Example
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["npm", "start"]
```

```dockerfile Multi-Stage Build
# Build stage
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Production stage
FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY package*.json ./
RUN npm install --production
EXPOSE 3000
CMD ["node", "dist/index.js"]
```
</CodeGroup>

### 2. Docker Image

A lightweight, standalone package that includes everything needed to run your application:
- Code
- Runtime
- Libraries
- Dependencies
- Environment variables

Images are built from a Dockerfile and can be versioned for consistent deployments.

### 3. Docker Container

A running instance of a Docker image. Containers are:
- Isolated from each other and the host system
- Lightweight and fast to start
- Portable across environments
- Ephemeral by design

<CodeGroup>
```bash Run Container
docker run -d -p 8080:80 --name my-app nginx
```

```bash Container Management
docker ps                    # List running containers
docker ps -a                 # List all containers
docker stop my-app           # Stop container
docker start my-app          # Start container
docker restart my-app        # Restart container
docker rm my-app             # Remove container
```
</CodeGroup>

### 4. Docker Registry

A centralized repository for storing and distributing Docker images.

<CodeGroup>
```bash Docker Hub
# Pull from Docker Hub
docker pull nginx:latest

# Push to Docker Hub
docker tag my-app:latest username/my-app:latest
docker push username/my-app:latest
```

```bash Private Registry
# Pull from private registry
docker pull registry.company.com/my-app:latest

# Login to private registry
docker login registry.company.com
```
</CodeGroup>

### 5. Docker Volumes

A way to persist data generated by containers. Volumes are:
- Outside the container's file system
- Shared between multiple containers
- Persistent across container restarts

<CodeGroup>
```bash Volume Management
# Create volume
docker volume create my-data

# List volumes
docker volume ls

# Inspect volume
docker volume inspect my-data

# Remove volume
docker volume rm my-data
```

```bash Using Volumes
# Mount volume
docker run -v my-data:/app/data my-app

# Bind mount
docker run -v /host/path:/container/path my-app

# Read-only mount
docker run -v my-data:/app/data:ro my-app
```
</CodeGroup>

### 6. Docker Compose

A tool for defining and running multi-container Docker applications.

<CodeGroup>
```yaml docker-compose.yml
version: '3.8'

services:
  web:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
    depends_on:
      - db
    volumes:
      - ./logs:/app/logs
  
  db:
    image: postgres:14-alpine
    environment:
      - POSTGRES_PASSWORD=secret
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  db-data:
```

```bash Compose Commands
docker-compose up -d         # Start services
docker-compose down          # Stop services
docker-compose logs -f       # View logs
docker-compose ps            # List services
docker-compose restart       # Restart services
```
</CodeGroup>

### 7. Docker Networks

Used to enable communication between containers and the host system.

<CodeGroup>
```bash Network Types
# Bridge (default)
docker network create my-network

# Host network
docker run --network host my-app

# None (no networking)
docker run --network none my-app
```

```bash Network Management
docker network ls
docker network inspect my-network
docker network connect my-network my-container
docker network disconnect my-network my-container
```
</CodeGroup>

### 8. Docker CLI

The primary way to interact with Docker.

<CodeGroup>
```bash Image Commands
docker images                # List images
docker build -t my-app .     # Build image
docker rmi my-app            # Remove image
docker image prune           # Remove unused images
```

```bash Container Commands
docker run [OPTIONS] IMAGE
docker exec -it container bash
docker logs -f container
docker inspect container
```

```bash System Commands
docker system df             # Show disk usage
docker system prune          # Remove unused data
docker stats                 # Show resource usage
```
</CodeGroup>

## Docker Best Practices

![Docker Best Practices](https://assets.bytebytego.com/diagrams/0016-9-docker-best-practices-you-must-know.png)

### 1. Use Official Images

This ensures security, reliability, and regular updates.

<CodeGroup>
```dockerfile Good
FROM node:18-alpine
```

```dockerfile Avoid
FROM some-random-user/node
```
</CodeGroup>

<Note>
Official images are maintained by the Docker community or the software vendor and undergo regular security scanning.
</Note>

### 2. Use Specific Image Versions

The default `latest` tag is unpredictable and causes unexpected behavior.

<CodeGroup>
```dockerfile Good - Specific Version
FROM node:18.17.0-alpine3.18
```

```dockerfile Avoid - Latest Tag
FROM node:latest
```
</CodeGroup>

### 3. Multi-Stage Builds

Reduces final image size by excluding build tools and dependencies.

<CodeGroup>
```dockerfile Multi-Stage Example
# Build stage
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o main .

# Production stage
FROM alpine:3.18
WORKDIR /app
COPY --from=builder /app/main .
EXPOSE 8080
CMD ["./main"]
```

```text Benefits
✓ Smaller final image (few MBs vs GBs)
✓ No build tools in production
✓ Faster deployment times
✓ Reduced attack surface
```
</CodeGroup>

### 4. Use .dockerignore

Excludes unnecessary files, speeds up builds, and reduces image size.

<CodeGroup>
```text .dockerignore
node_modules
npm-debug.log
.git
.gitignore
.env
*.md
.DS_Store
tests
*.test.js
coverage
```

```text Benefits
✓ Faster build times
✓ Smaller images
✓ No sensitive files in image
✓ Cleaner context
```
</CodeGroup>

### 5. Use the Least Privileged User

Enhances security by limiting container privileges.

<CodeGroup>
```dockerfile Create User
FROM node:18-alpine

# Create app user
RUN addgroup -g 1001 -S appuser && \
    adduser -S appuser -u 1001

WORKDIR /app
COPY --chown=appuser:appuser . .

# Switch to non-root user
USER appuser

CMD ["node", "index.js"]
```

```text Why It Matters
✓ Prevents privilege escalation
✓ Limits damage from vulnerabilities
✓ Follows principle of least privilege
✓ Compliance requirement
```
</CodeGroup>

### 6. Use Environment Variables

Increases flexibility and portability across different environments.

<CodeGroup>
```dockerfile In Dockerfile
FROM node:18-alpine
WORKDIR /app
COPY . .

ENV NODE_ENV=production
ENV PORT=3000

EXPOSE $PORT
CMD ["node", "index.js"]
```

```bash At Runtime
# Single variable
docker run -e NODE_ENV=development my-app

# Multiple variables
docker run \
  -e NODE_ENV=production \
  -e DB_HOST=postgres \
  -e DB_PORT=5432 \
  my-app

# From file
docker run --env-file .env my-app
```
</CodeGroup>

### 7. Order Matters for Caching

Order your steps from least to most frequently changing to optimize caching.

<CodeGroup>
```dockerfile Optimized
FROM node:18-alpine
WORKDIR /app

# Dependencies change less frequently
COPY package*.json ./
RUN npm install

# Code changes more frequently
COPY . .

CMD ["npm", "start"]
```

```dockerfile Not Optimized
FROM node:18-alpine
WORKDIR /app

# Code copied first - breaks cache often
COPY . .
RUN npm install

CMD ["npm", "start"]
```
</CodeGroup>

### 8. Label Your Images

Improves organization and helps with image management.

<CodeGroup>
```dockerfile With Labels
FROM node:18-alpine

LABEL maintainer="dev@company.com"
LABEL version="1.0"
LABEL description="My web application"
LABEL org.opencontainers.image.source="https://github.com/company/repo"

WORKDIR /app
COPY . .
CMD ["node", "index.js"]
```

```bash Filter by Labels
# Find images by label
docker images --filter "label=version=1.0"

# Inspect labels
docker inspect --format='{{json .Config.Labels}}' my-app
```
</CodeGroup>

### 9. Scan Images for Vulnerabilities

Find security vulnerabilities before they become bigger problems.

<CodeGroup>
```bash Docker Scan
# Scan local image
docker scan my-app:latest

# Scan with severity filter
docker scan --severity high my-app:latest
```

```bash Trivy Scanner
# Install Trivy
brew install aquasecurity/trivy/trivy

# Scan image
trivy image my-app:latest

# Scan with severity filter
trivy image --severity HIGH,CRITICAL my-app:latest
```

```bash Snyk
# Install Snyk
npm install -g snyk

# Authenticate
snyk auth

# Scan image
snyk container test my-app:latest
```
</CodeGroup>

## Additional Best Practices

### Minimize Layer Count

<CodeGroup>
```dockerfile Good - Combined Commands
RUN apt-get update && \
    apt-get install -y curl vim && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

```dockerfile Avoid - Separate Commands
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y vim
RUN apt-get clean
```
</CodeGroup>

### Use Health Checks

<CodeGroup>
```dockerfile Health Check
FROM node:18-alpine
WORKDIR /app
COPY . .

HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD node healthcheck.js || exit 1

CMD ["node", "index.js"]
```

```bash Check Health Status
docker ps
# Shows health status in STATUS column

docker inspect --format='{{.State.Health.Status}}' container-name
```
</CodeGroup>

### Optimize for Production

<Steps>
  <Step title="Use Alpine Images">
    Alpine-based images are smaller and have fewer vulnerabilities
  </Step>
  
  <Step title="Remove Build Dependencies">
    Use multi-stage builds to exclude build tools
  </Step>
  
  <Step title="Set Resource Limits">
    Use `--memory` and `--cpus` flags to limit resource usage
  </Step>
  
  <Step title="Enable Read-Only Root Filesystem">
    Run containers with `--read-only` flag when possible
  </Step>
</Steps>

## Common Docker Commands Reference

### Container Management

<CodeGroup>
```bash Lifecycle
docker create [IMAGE]        # Create container
docker start [CONTAINER]     # Start container
docker stop [CONTAINER]      # Stop container
docker restart [CONTAINER]   # Restart container
docker pause [CONTAINER]     # Pause container
docker unpause [CONTAINER]   # Unpause container
docker kill [CONTAINER]      # Kill container
docker rm [CONTAINER]        # Remove container
```

```bash Inspection
docker ps                    # Running containers
docker ps -a                 # All containers
docker logs [CONTAINER]      # View logs
docker inspect [CONTAINER]   # Detailed info
docker stats                 # Resource usage
docker top [CONTAINER]       # Running processes
```
</CodeGroup>

### Image Management

<CodeGroup>
```bash Build & Push
docker build -t name:tag .   # Build image
docker tag SOURCE TARGET     # Tag image
docker push name:tag         # Push image
docker pull name:tag         # Pull image
```

```bash Cleanup
docker rmi [IMAGE]           # Remove image
docker image prune           # Remove unused images
docker image prune -a        # Remove all unused images
```
</CodeGroup>

Docker is powerful but requires understanding of best practices for security, performance, and maintainability. Start simple and gradually adopt advanced patterns.
