## Database Data Structures

<img src="https://assets.bytebytego.com/diagrams/0181-eight-ds-db.jpg" alt="Database Data Structures" />

The choice of data structure will vary depending on your use case. Data can be indexed in memory or on disk. Data formats vary, such as numbers, strings, geographic coordinates, etc. The system might be write-heavy or read-heavy. All of these factors affect your choice of database index format.

### Popular Database Index Structures

<AccordionGroup>
  <Accordion title="Skiplist" icon="bars-staggered">
    A common in-memory index type.
    
    **Characteristics:**
    - Probabilistic data structure
    - Multiple levels of linked lists
    - O(log n) search, insert, delete
    - Simpler than balanced trees
    
    **Used in:**
    - Redis (sorted sets)
    - In-memory databases
    - Concurrent data structures
    
    **Trade-offs:**
    - ✅ Simple implementation
    - ✅ Good concurrency
    - ❌ Extra memory for pointers
    - ❌ Probabilistic guarantees
  </Accordion>

  <Accordion title="Hash Index" icon="hashtag">
    A very common implementation of the "Map" data structure (or "Collection").
    
    **Characteristics:**
    - Direct addressing
    - O(1) average lookup
    - Hash function + buckets
    - Collision handling
    
    **Used in:**
    - Key-value stores
    - Database hash indexes
    - Cache implementations
    
    **Trade-offs:**
    - ✅ Fast lookups
    - ✅ Fast inserts
    - ❌ No range queries
    - ❌ No ordered traversal
  </Accordion>

  <Accordion title="SSTable" icon="table">
    Immutable on-disk "Map" implementation.
    
    **Characteristics:**
    - Sorted Strings Table
    - Immutable on disk
    - Sorted by key
    - Efficient merging
    
    **Used in:**
    - Cassandra
    - LevelDB
    - RocksDB
    
    **Trade-offs:**
    - ✅ Sequential writes
    - ✅ Efficient merging
    - ❌ Write amplification
    - ❌ Read overhead
  </Accordion>

  <Accordion title="LSM Tree" icon="tree">
    Skiplist + SSTable. High write throughput.
    
    **Characteristics:**
    - Log-Structured Merge Tree
    - In-memory + on-disk components
    - Background compaction
    - Write-optimized
    
    **Used in:**
    - Cassandra
    - LevelDB
    - RocksDB
    - HBase
    
    **Trade-offs:**
    - ✅ High write throughput
    - ✅ Efficient updates
    - ❌ Read amplification
    - ❌ Compaction overhead
  </Accordion>

  <Accordion title="B-tree" icon="diagram-project">
    Disk-based solution. Consistent read/write performance.
    
    **Characteristics:**
    - Self-balancing tree
    - Multiple keys per node
    - Optimized for disk I/O
    - Range queries supported
    
    **Used in:**
    - MySQL (InnoDB)
    - PostgreSQL
    - Most relational databases
    - File systems
    
    **Trade-offs:**
    - ✅ Balanced read/write
    - ✅ Range queries
    - ✅ Sorted order
    - ❌ Write overhead (rebalancing)
  </Accordion>

  <Accordion title="Inverted Index" icon="magnifying-glass">
    Used for document indexing.
    
    **Characteristics:**
    - Term → Document mapping
    - Full-text search
    - Posting lists
    - Token-based
    
    **Used in:**
    - Elasticsearch
    - Apache Lucene
    - Search engines
    
    **Trade-offs:**
    - ✅ Fast text search
    - ✅ Ranking support
    - ❌ Storage overhead
    - ❌ Update complexity
  </Accordion>

  <Accordion title="Suffix Tree" icon="code-merge">
    For string pattern search.
    
    **Characteristics:**
    - All suffixes indexed
    - Pattern matching O(m)
    - Space-intensive
    - Build time O(n)
    
    **Used in:**
    - Bioinformatics
    - String search algorithms
    - Text processing
    
    **Trade-offs:**
    - ✅ Fast pattern matching
    - ✅ Multiple patterns
    - ❌ High memory usage
    - ❌ Complex implementation
  </Accordion>

  <Accordion title="R-tree" icon="map">
    Multi-dimension search, such as finding the nearest neighbor.
    
    **Characteristics:**
    - Spatial indexing
    - Bounding boxes (MBR)
    - Multi-dimensional data
    - Range queries
    
    **Used in:**
    - PostGIS
    - Geospatial databases
    - Map applications
    
    **Trade-offs:**
    - ✅ Spatial queries
    - ✅ Nearest neighbor
    - ❌ Complex updates
    - ❌ Overlap in nodes
  </Accordion>
</AccordionGroup>

<Warning>
  This is not an exhaustive list of all database index types. The choice depends on your specific use case, access patterns, and performance requirements.
</Warning>

## B-Tree vs LSM-Tree

<img src="https://assets.bytebytego.com/diagrams/0091-btree-lsm.png" alt="B-Tree vs LSM-Tree" />

Two of the most important data structures in database design:

<Tabs>
  <Tab title="B-Tree">
    ### Most Widely Used in Relational Databases
    
    B-Tree is the most widely used indexing data structure in almost all relational databases.
    
    **How it Works:**
    
    The basic unit of information storage in B-Tree is usually called a "page". Looking up a key traces down the range of keys until the actual value is found.
    
    **Characteristics:**
    
    - Self-balancing tree structure
    - Multiple keys per node
    - All leaves at same level
    - Sorted key order
    - Range queries efficient
    
    **Performance:**
    
    - Read: O(log n) - **Optimized for reads**
    - Write: O(log n)
    - Range query: Efficient
    - Point query: Efficient
    
    **Best For:**
    
    - Read-heavy workloads
    - Range queries
    - Transactional databases
    - Traditional RDBMS
    
    **Used In:**
    
    - MySQL InnoDB
    - PostgreSQL
    - SQLite
    - Most relational databases
  </Tab>
  
  <Tab title="LSM-Tree">
    ### Popular in NoSQL Databases
    
    LSM-Tree (Log-Structured Merge Tree) is widely used by many NoSQL databases.
    
    **How it Works:**
    
    LSM-trees maintain key-value pairs and are persisted to disk using a Sorted Strings Table (SSTable), in which the keys are sorted.
    
    Level 0 segments are periodically merged into Level 1 segments. This process is called **compaction**.
    
    **Characteristics:**
    
    - Write to in-memory structure first
    - Flush to disk as SSTables
    - Background compaction
    - Multiple levels
    - Append-only writes
    
    **Performance:**
    
    - Write: O(1) amortized - **Optimized for writes**
    - Read: O(log n) with overhead
    - Compaction: Background process
    - Space amplification: Higher
    
    **Best For:**
    
    - Write-heavy workloads
    - Time-series data
    - Logging systems
    - High-throughput applications
    
    **Used In:**
    
    - Cassandra
    - RocksDB
    - LevelDB
    - HBase
  </Tab>
  
  <Tab title="Comparison">
    ### The Key Difference
    
    <CardGroup cols={2}>
      <Card title="B-Tree: Read-Optimized" icon="book-open">
        **Faster Reads**
        
        - Fewer disk seeks
        - Predictable performance
        - Better for point queries
        - Efficient range scans
      </Card>
      
      <Card title="LSM-Tree: Write-Optimized" icon="pen">
        **Faster Writes**
        
        - Sequential writes
        - Less write amplification initially
        - Better write throughput
        - Batch operations efficient
      </Card>
    </CardGroup>
    
    ### Trade-offs
    
    | Aspect | B-Tree | LSM-Tree |
    |--------|--------|----------|
    | **Read Performance** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
    | **Write Performance** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
    | **Space Efficiency** | ⭐⭐⭐⭐ | ⭐⭐⭐ |
    | **Range Queries** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
    | **Point Queries** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
    | **Write Amplification** | Higher | Lower (initially) |
    | **Read Amplification** | Lower | Higher |
    | **Compaction Overhead** | Lower | Higher |
    
    ### When to Choose
    
    **Choose B-Tree when:**
    - Read-heavy workload
    - Need consistent read latency
    - Range queries are common
    - Disk space is limited
    
    **Choose LSM-Tree when:**
    - Write-heavy workload
    - Can tolerate read latency variance
    - Sequential writes common
    - Compaction overhead acceptable
  </Tab>
</Tabs>

## Data Structure Selection Guide

### Quick Reference

<AccordionGroup>
  <Accordion title="Need Fast Lookups?" icon="magnifying-glass">
    **Hash Table / Hash Map**
    
    - O(1) average case
    - Key-value pairs
    - No ordering
    
    **When to use:**
    - Caching
    - Deduplication
    - Fast membership testing
    - Counting occurrences
  </Accordion>

  <Accordion title="Need Ordering?" icon="arrow-down-1-9">
    **Tree-based structures**
    
    - B-Tree: Disk-based, balanced
    - BST: In-memory, may be unbalanced
    - Red-Black Tree: Self-balancing
    
    **When to use:**
    - Range queries
    - Sorted iteration
    - Database indexes
    - Priority queues (Heap)
  </Accordion>

  <Accordion title="Need Fast Writes?" icon="pen">
    **LSM-Tree or Log-based**
    
    - Append-only writes
    - Background compaction
    - Write-optimized
    
    **When to use:**
    - Time-series data
    - Logging systems
    - Write-heavy workloads
    - Event sourcing
  </Accordion>

  <Accordion title="Need Spatial Queries?" icon="map">
    **R-Tree or QuadTree**
    
    - Multi-dimensional indexing
    - Bounding boxes
    - Nearest neighbor
    
    **When to use:**
    - Geospatial applications
    - Location-based services
    - Map applications
    - Collision detection
  </Accordion>

  <Accordion title="Need Text Search?" icon="magnifying-glass-plus">
    **Inverted Index or Trie**
    
    - Term-based indexing
    - Full-text search
    - Prefix matching
    
    **When to use:**
    - Search engines
    - Autocomplete
    - Document search
    - Log analysis
  </Accordion>

  <Accordion title="Need Graph Relationships?" icon="diagram-project">
    **Graph structures**
    
    - Adjacency list
    - Adjacency matrix
    - Graph databases
    
    **When to use:**
    - Social networks
    - Recommendation engines
    - Path finding
    - Dependency resolution
  </Accordion>
</AccordionGroup>

## Time Complexity Cheat Sheet

### Common Data Structure Operations

| Data Structure | Access | Search | Insert | Delete | Space |
|---------------|--------|--------|--------|--------|-------|
| **Array** | O(1) | O(n) | O(n) | O(n) | O(n) |
| **Linked List** | O(n) | O(n) | O(1) | O(1) | O(n) |
| **Hash Table** | - | O(1)* | O(1)* | O(1)* | O(n) |
| **Binary Search Tree** | O(log n)* | O(log n)* | O(log n)* | O(log n)* | O(n) |
| **B-Tree** | O(log n) | O(log n) | O(log n) | O(log n) | O(n) |
| **Skip List** | O(log n) | O(log n) | O(log n) | O(log n) | O(n log n) |
| **Heap** | - | O(n) | O(log n) | O(log n) | O(n) |

<Note>
  * Average case. Worst case may differ.
</Note>

## Interview Tips

<Warning>
  **Common Mistakes**
  
  - Choosing hash table when ordering is needed
  - Using array when frequent insertions/deletions required
  - Not considering space complexity
  - Ignoring the read/write ratio of the workload
  - Forgetting about the cost of data structure maintenance (rebalancing, compaction)
</Warning>

<Tip>
  **Interview Strategy**
  
  When discussing data structures in system design:
  
  1. **Understand the access patterns**: Read-heavy? Write-heavy? Mixed?
  2. **Consider the data characteristics**: Size? Growth rate? Data types?
  3. **Think about trade-offs**: Speed vs space? Consistency vs performance?
  4. **Discuss real-world examples**: How do major companies solve similar problems?
  5. **Consider maintenance**: What's the operational overhead?
</Tip>

## Practice Problems

<AccordionGroup>
  <Accordion title="Design a Cache System" icon="database">
    **Consider:**
    - Hash table for O(1) lookups
    - Doubly linked list for LRU eviction
    - Combined: LRU Cache
    
    **Trade-offs:**
    - Memory overhead for linked list
    - Fast reads and updates
    - Eviction policy implementation
  </Accordion>

  <Accordion title="Design a Real-time Leaderboard" icon="trophy">
    **Consider:**
    - Sorted Set (Skip List) for ranking
    - Hash table for user scores
    - Redis sorted sets
    
    **Trade-offs:**
    - Memory for maintaining order
    - Fast score updates
    - Efficient range queries for top N
  </Accordion>

  <Accordion title="Design a Geospatial Index" icon="map-location-dot">
    **Consider:**
    - QuadTree for 2D partitioning
    - Geohash for encoding
    - R-Tree for bounding boxes
    
    **Trade-offs:**
    - Balance between precision and performance
    - Update complexity
    - Query efficiency
  </Accordion>

  <Accordion title="Design a Search Engine" icon="magnifying-glass">
    **Consider:**
    - Inverted index for terms
    - Trie for autocomplete
    - Ranking algorithms
    
    **Trade-offs:**
    - Storage overhead
    - Index update latency
    - Query performance
  </Accordion>
</AccordionGroup>
