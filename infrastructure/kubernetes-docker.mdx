## Docker Fundamentals

Docker is a platform for developing, shipping, and running applications in containers. Containers package an application with all its dependencies, ensuring consistency across different environments.

<Frame>
  <img src="https://assets.bytebytego.com/diagrams/0414-how-does-docker-work.png" alt="How Docker Works" />
</Frame>

### Docker Architecture

Docker has three main components that work together:

<CardGroup cols={3}>
  <Card title="Docker Client" icon="terminal">
    The command-line interface that users interact with. The docker client talks to the Docker daemon
  </Card>
  
  <Card title="Docker Host" icon="server">
    The Docker daemon listens for Docker API requests and manages Docker objects such as images, containers, networks, and volumes
  </Card>
  
  <Card title="Docker Registry" icon="database">
    A Docker registry stores Docker images. Docker Hub is a public registry that anyone can use
  </Card>
</CardGroup>

### How Docker Run Works

When you execute `docker run`, the following sequence occurs:

<Steps>
  <Step title="Image Pull">
    Docker pulls the image from the registry (if not already available locally)
  </Step>
  
  <Step title="Container Creation">
    Docker creates a new container from the image
  </Step>
  
  <Step title="Filesystem Allocation">
    Docker allocates a read-write filesystem to the container
  </Step>
  
  <Step title="Network Setup">
    Docker creates a network interface to connect the container to the default network
  </Step>
  
  <Step title="Container Start">
    Docker starts the container and executes the specified command
  </Step>
</Steps>

## Essential Docker Concepts

<Frame>
  <img src="https://assets.bytebytego.com/diagrams/0012-8-must-know-docker-concepts.png" alt="8 Must-Know Docker Concepts" />
</Frame>

<AccordionGroup>
  <Accordion title="Dockerfile" icon="file-code">
    Contains the instructions to build a Docker image by specifying the base image, dependencies, and run command.
    
    ```dockerfile
    FROM node:18-alpine
    WORKDIR /app
    COPY package*.json ./
    RUN npm install
    COPY . .
    EXPOSE 3000
    CMD ["npm", "start"]
    ```
  </Accordion>
  
  <Accordion title="Docker Image" icon="box-archive">
    A lightweight, standalone package that includes everything (code, libraries, and dependencies) needed to run your application. Images are built from a Dockerfile and can be versioned.
    
    **Key Characteristics:**
    - Immutable and portable
    - Layered architecture for efficiency
    - Can be versioned with tags
    - Stored in registries (Docker Hub, ECR, etc.)
  </Accordion>
  
  <Accordion title="Docker Container" icon="cube">
    A running instance of a Docker image. Containers are isolated from each other and the host system, providing a secure and reproducible environment for running your apps.
    
    **Key Characteristics:**
    - Lightweight and fast to start
    - Isolated process and filesystem
    - Ephemeral by default
    - Can be paused, stopped, and restarted
  </Accordion>
  
  <Accordion title="Docker Registry" icon="warehouse">
    A centralized repository for storing and distributing Docker images. For example, Docker Hub is the default public registry but you can also set up private registries.
    
    **Popular Registries:**
    - Docker Hub (public)
    - Amazon ECR
    - Google Container Registry
    - Azure Container Registry
  </Accordion>
  
  <Accordion title="Docker Volumes" icon="hard-drive">
    A way to persist data generated by containers. Volumes are outside the container's file system and can be shared between multiple containers.
    
    **Benefits:**
    - Data persists after container removal
    - Can be shared between containers
    - Managed independently of containers
    - Better performance than bind mounts
  </Accordion>
  
  <Accordion title="Docker Compose" icon="layer-group">
    A tool for defining and running multi-container Docker applications, making it easy to manage the entire stack.
    
    ```yaml
    version: '3.8'
    services:
      web:
        build: .
        ports:
          - "3000:3000"
      db:
        image: postgres:14
        environment:
          POSTGRES_PASSWORD: secret
    ```
  </Accordion>
  
  <Accordion title="Docker Networks" icon="network-wired">
    Used to enable communication between containers and the host system. Custom networks can isolate containers or enable selective communication.
    
    **Network Types:**
    - Bridge (default)
    - Host
    - Overlay (for Swarm)
    - None
  </Accordion>
  
  <Accordion title="Docker CLI" icon="terminal">
    The primary way to interact with Docker, providing commands for building images, running containers, managing volumes, and performing other operations.
  </Accordion>
</AccordionGroup>

## Docker Best Practices

<Frame>
  <img src="https://assets.bytebytego.com/diagrams/0016-9-docker-best-practices-you-must-know.png" alt="9 Docker Best Practices" />
</Frame>

<CardGroup cols={2}>
  <Card title="1. Use Official Images" icon="circle-check">
    This ensures security, reliability, and regular updates from trusted sources
  </Card>
  
  <Card title="2. Use Specific Image Versions" icon="tag">
    The default `latest` tag is unpredictable and causes unexpected behavior. Always specify version numbers
    
    ```dockerfile
    # Bad
    FROM node:latest
    
    # Good
    FROM node:18.17.0-alpine
    ```
  </Card>
  
  <Card title="3. Multi-Stage Builds" icon="layer-group">
    Reduces final image size by excluding build tools and dependencies
    
    ```dockerfile
    # Build stage
    FROM node:18 AS builder
    WORKDIR /app
    COPY . .
    RUN npm ci && npm run build
    
    # Production stage
    FROM node:18-alpine
    WORKDIR /app
    COPY --from=builder /app/dist ./dist
    CMD ["node", "dist/index.js"]
    ```
  </Card>
  
  <Card title="4. Use .dockerignore" icon="file-slash">
    Excludes unnecessary files, speeds up builds, and reduces image size
    
    ```
    node_modules
    .git
    .env
    *.log
    ```
  </Card>
  
  <Card title="5. Use Least Privileged User" icon="user-shield">
    Enhances security by limiting container privileges
    
    ```dockerfile
    FROM node:18-alpine
    RUN addgroup -S appgroup && adduser -S appuser -G appgroup
    USER appuser
    ```
  </Card>
  
  <Card title="6. Use Environment Variables" icon="gear">
    Increases flexibility and portability across different environments
    
    ```dockerfile
    ENV NODE_ENV=production
    ENV PORT=3000
    ```
  </Card>
  
  <Card title="7. Order Matters for Caching" icon="arrows-up-down">
    Order your steps from least to most frequently changing to optimize caching
    
    ```dockerfile
    # Dependencies change less frequently
    COPY package*.json ./
    RUN npm ci
    
    # Code changes more frequently
    COPY . .
    ```
  </Card>
  
  <Card title="8. Label Your Images" icon="tags">
    It improves organization and helps with image management
    
    ```dockerfile
    LABEL maintainer="team@example.com"
    LABEL version="1.0"
    LABEL description="Production API service"
    ```
  </Card>
  
  <Card title="9. Scan Images for Vulnerabilities" icon="shield-halved">
    Find security vulnerabilities before they become bigger problems
    
    ```bash
    docker scan myimage:latest
    ```
  </Card>
</CardGroup>

## Kubernetes Overview

<Frame>
  <img src="https://assets.bytebytego.com/diagrams/0245-k8s.png" alt="Kubernetes Architecture" />
</Frame>

Kubernetes (k8s) is a container orchestration system used for container deployment and management. Its design is greatly impacted by Google's internal system Borg.

A Kubernetes cluster consists of a set of worker machines, called nodes, that run containerized applications. Every cluster has at least one worker node. The worker node(s) host the Pods that are the components of the application workload. The control plane manages the worker nodes and the Pods in the cluster.

<Note>
In production environments, the control plane usually runs across multiple computers and a cluster usually runs multiple nodes, providing fault tolerance and high availability.
</Note>

### Control Plane Components

<AccordionGroup>
  <Accordion title="API Server" icon="server">
    The API server talks to all the components in the Kubernetes cluster. All the operations on pods are executed by talking to the API server.
    
    **Key Functions:**
    - Exposes the Kubernetes API
    - Front-end for the control plane
    - Validates and processes REST requests
    - Updates etcd with cluster state
  </Accordion>
  
  <Accordion title="Scheduler" icon="calendar-check">
    The scheduler watches the workloads on pods and assigns loads on newly created pods.
    
    **Responsibilities:**
    - Selects optimal nodes for pods
    - Considers resource requirements
    - Respects constraints and affinity rules
    - Balances workload distribution
  </Accordion>
  
  <Accordion title="Controller Manager" icon="sliders">
    The controller manager runs the controllers, including:
    - **Node Controller**: Monitors node health
    - **Job Controller**: Manages one-off tasks
    - **EndpointSlice Controller**: Manages network endpoints
    - **ServiceAccount Controller**: Manages service accounts
  </Accordion>
  
  <Accordion title="etcd" icon="database">
    etcd is a key-value store used as Kubernetes' backing store for all cluster data.
    
    **Characteristics:**
    - Distributed and highly available
    - Stores cluster configuration and state
    - Source of truth for the cluster
    - Should be backed up regularly
  </Accordion>
</AccordionGroup>

### Node Components

<AccordionGroup>
  <Accordion title="Pods" icon="cube">
    A pod is a group of containers and is the smallest unit that Kubernetes administers. Pods have a single IP address applied to every container within the pod.
    
    **Key Concepts:**
    - Shared network namespace
    - Shared storage volumes
    - Ephemeral by nature
    - Scaled through ReplicaSets
  </Accordion>
  
  <Accordion title="Kubelet" icon="gear">
    An agent that runs on each node in the cluster. It ensures containers are running in a Pod.
    
    **Responsibilities:**
    - Monitors pod specifications
    - Ensures containers are healthy
    - Reports node and pod status
    - Manages container lifecycle
  </Accordion>
  
  <Accordion title="Kube Proxy" icon="network-wired">
    kube-proxy is a network proxy that runs on each node in your cluster. It routes traffic coming into a node from the service. It forwards requests for work to the correct containers.
    
    **Functions:**
    - Maintains network rules
    - Enables service abstraction
    - Load balances traffic
    - Implements service networking
  </Accordion>
</AccordionGroup>

## Kubernetes Service Types

<Frame>
  <img src="https://assets.bytebytego.com/diagrams/0005-4-k8s-service-types.png" alt="Kubernetes Service Types" />
</Frame>

In Kubernetes, a Service is a method for exposing a network application in the cluster. The "type" property in the Service's specification determines how the service is exposed to the network.

<CardGroup cols={2}>
  <Card title="ClusterIP" icon="circle-nodes">
    **Default service type**. Kubernetes assigns a cluster-internal IP address to ClusterIP service. This makes the service only reachable within the cluster.
    
    **Use Cases:**
    - Internal microservices communication
    - Backend services not exposed externally
    - Database connections within cluster
  </Card>
  
  <Card title="NodePort" icon="door-open">
    Exposes the service outside of the cluster by adding a cluster-wide port on top of ClusterIP. We can request the service by `NodeIP:NodePort`.
    
    **Use Cases:**
    - Development and testing
    - Direct access to services
    - Port range: 30000-32767
  </Card>
  
  <Card title="LoadBalancer" icon="scale-balanced">
    Exposes the Service externally using a cloud provider's load balancer.
    
    **Use Cases:**
    - Production external services
    - Cloud-native applications
    - Automatic load balancing
    - Integrates with AWS ELB, GCP Load Balancer, etc.
  </Card>
  
  <Card title="ExternalName" icon="arrow-up-right-from-square">
    Maps a Service to a domain name. This is commonly used to create a service within Kubernetes to represent an external database.
    
    **Use Cases:**
    - External database connections
    - Third-party API integration
    - Service migration scenarios
  </Card>
</CardGroup>

## Kubernetes Deployment Strategies

<Frame>
  <img src="https://assets.bytebytego.com/diagrams/0247-kubernates-deployment-strategy.jpeg" alt="Kubernetes Deployment Strategies" />
</Frame>

Each strategy offers a unique approach to manage updates:

<AccordionGroup>
  <Accordion title="Recreate" icon="rotate">
    All existing instances are terminated at once, and new instances with the updated version are created.
    
    - **Downtime:** Yes
    - **Use case:** Non-critical applications or during initial development stages
    - **Risk:** High
    - **Speed:** Fast
  </Accordion>
  
  <Accordion title="Rolling Update" icon="arrows-rotate">
    Application instances are updated one by one, ensuring high availability during the process.
    
    - **Downtime:** No
    - **Use case:** Periodic releases
    - **Risk:** Low to medium
    - **Speed:** Medium
    
    This is the default Kubernetes deployment strategy.
  </Accordion>
  
  <Accordion title="Shadow" icon="ghost">
    A copy of the live traffic is redirected to the new version for testing without affecting production users.
    
    - **Downtime:** No
    - **Use case:** Validating new version performance and behavior in a real environment
    - **Risk:** Low
    - **Complexity:** Very high
    
    This is the most complex deployment strategy and involves establishing mock services to interact with the new version of the deployment.
  </Accordion>
  
  <Accordion title="Canary" icon="bird">
    The new version is released to a subset of users or servers for testing before broader deployment.
    
    - **Downtime:** No
    - **Use case:** Impact validation on a subset of users
    - **Risk:** Low
    - **Speed:** Gradual
    
    Typically starts with 5-10% of traffic, then gradually increases.
  </Accordion>
  
  <Accordion title="Blue-Green" icon="swap">
    Two identical environments are maintained: one with the current version (blue) and the other with the updated version (green). Traffic starts with blue, then switches to the prepared green environment for the updated version.
    
    - **Downtime:** No
    - **Use case:** High-stake updates
    - **Risk:** Low (easy rollback)
    - **Cost:** High (requires double resources)
  </Accordion>
  
  <Accordion title="A/B Testing" icon="chart-line">
    Multiple versions are concurrently tested on different users to compare performance or user experience.
    
    - **Downtime:** Not directly applicable
    - **Use case:** Optimizing user experience
    - **Duration:** Extended (weeks to months)
    - **Focus:** Business metrics and user behavior
  </Accordion>
</AccordionGroup>

## Kubernetes Command Cheatsheet

<Frame>
  <img src="https://assets.bytebytego.com/diagrams/0248-kubernetes-command-cheatsheet.png" alt="Kubernetes Command Cheatsheet" />
</Frame>

Kubernetes is an open-source container orchestration platform that automates the deployment, scaling, and management of containerized applications. Initially developed by Google, Kubernetes is now maintained by CNCF (Cloud Native Computing Foundation).

### Essential Commands by Category

<Tabs>
  <Tab title="Cluster Management">
    ```bash
    # View cluster information
    kubectl cluster-info
    
    # View nodes
    kubectl get nodes
    
    # Describe node details
    kubectl describe node <node-name>
    
    # View cluster events
    kubectl get events
    ```
  </Tab>
  
  <Tab title="Deployments">
    ```bash
    # Create deployment
    kubectl create deployment <name> --image=<image>
    
    # Get deployments
    kubectl get deployments
    
    # Scale deployment
    kubectl scale deployment <name> --replicas=3
    
    # Update image
    kubectl set image deployment/<name> <container>=<image>
    
    # Rollout status
    kubectl rollout status deployment/<name>
    
    # Rollback deployment
    kubectl rollout undo deployment/<name>
    ```
  </Tab>
  
  <Tab title="Pods">
    ```bash
    # Get pods
    kubectl get pods
    
    # Describe pod
    kubectl describe pod <pod-name>
    
    # View pod logs
    kubectl logs <pod-name>
    
    # Execute command in pod
    kubectl exec -it <pod-name> -- /bin/bash
    
    # Delete pod
    kubectl delete pod <pod-name>
    ```
  </Tab>
  
  <Tab title="Services">
    ```bash
    # Get services
    kubectl get services
    
    # Expose deployment
    kubectl expose deployment <name> --type=LoadBalancer --port=80
    
    # Describe service
    kubectl describe service <service-name>
    
    # Delete service
    kubectl delete service <service-name>
    ```
  </Tab>
  
  <Tab title="ConfigMaps & Secrets">
    ```bash
    # Create configmap
    kubectl create configmap <name> --from-file=<path>
    
    # Get configmaps
    kubectl get configmaps
    
    # Create secret
    kubectl create secret generic <name> --from-literal=key=value
    
    # Get secrets
    kubectl get secrets
    ```
  </Tab>
</Tabs>
